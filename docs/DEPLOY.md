# Deploy

How to deploy `sensor_app` to [Render](https://render.com) on the **free
tier** from the version-controlled [`render.yaml`](../render.yaml) Blueprint.
For local runs see [PLAYBOOK.md](PLAYBOOK.md).

## What gets deployed

The Blueprint declares four resources:

| Resource   | Render type        | Notes                                                  |
| ---------- | ------------------ | ------------------------------------------------------ |
| `sensor-app-db`       | Managed PostgreSQL (free) | Schema applied by Alembic on backend boot.    |
| `sensor-app-backend`  | Docker web service (free) | gunicorn on `$PORT`, `DEBUG=false`.           |
| `sensor-app-frontend` | Static site (free)        | Angular SPA on the CDN; backend URL baked in. |
| `sensor-app-grafana`  | Docker web service (free) | Dashboards over the DB, provisioned as code.  |

```
[ Static site (CDN) ]  --HTTPS/JSON-->  [ Docker web service ]  -->  [ Managed Postgres ]
 sensor-app-frontend                       sensor-app-backend            sensor-app-db
 Angular SPA, API URL                       gunicorn + in-process              ^
 baked in at build                          generator (free-tier mode)         |
        |                                                                       |
        +-- "View charts" link -->  [ Docker web service ] --(reads, SSL)-------+
                                       sensor-app-grafana
                                       provisioned dashboards
```

The frontend calls the backend cross-origin, so the backend's `CORS_ORIGINS`
is scoped to the static site's URL. The frontend also links out to Grafana (a
plain navigation, no cross-origin API call), and Grafana reads the managed
Postgres directly over SSL.

## One-time setup

1. Push this repo to GitHub (already done: `braboj/demo-sensor-app`).
2. In the Render dashboard: **New +** → **Blueprint**.
3. Connect the repository and select the branch. Render reads `render.yaml`
   from the repo root and shows the four resources above.
4. Review and **Apply**. Render provisions the database, builds the backend
   image, runs the static-site build, and wires the environment variables
   declared in the Blueprint.
5. Wire up CI-triggered deploys (see "Deploy trigger" below). For each of the
   three web services, open its Render dashboard → **Settings** → **Deploy
   Hook**, copy the URL, and store it as a GitHub Actions secret:

   | Service               | GitHub secret                  |
   | --------------------- | ------------------------------ |
   | `sensor-app-backend`  | `RENDER_DEPLOY_HOOK_BACKEND`   |
   | `sensor-app-frontend` | `RENDER_DEPLOY_HOOK_FRONTEND`  |
   | `sensor-app-grafana`  | `RENDER_DEPLOY_HOOK_GRAFANA`   |

   Until a service's secret is set, its deploy step skips (CI stays green) and
   that service simply never redeploys.

`SECRET_KEY` is generated by Render (`generateValue`); `DATABASE_URL` is wired
from the managed database. No secrets live in the repo.

## Deploy trigger

Deploys are **CI-triggered, not push-based** (ADR-0009). Every service sets
`autoDeploy: false`, so Render does not deploy on its own GitHub webhook.
Instead the `deploy` job in
[`.github/workflows/ci.yml`](../.github/workflows/ci.yml) POSTs a Render deploy
hook per service — but only on `push` to `main` and only after the backend,
frontend, and gitleaks jobs pass. A red pipeline therefore never deploys, and
each deploy is an observable step in the Actions run.

```
push to main → backend + frontend + gitleaks pass → deploy job
                                                        │
                    POST RENDER_DEPLOY_HOOK_BACKEND ────┤
                    POST RENDER_DEPLOY_HOOK_FRONTEND ───┤
                    POST RENDER_DEPLOY_HOOK_GRAFANA ────┘
                                                        │
                                   Render rebuilds + redeploys each service
```

This replaces Render's push-based `autoDeploy`, which fired unreliably and
deployed regardless of CI status. Pull requests and forks skip the job. CodeQL
runs in a separate workflow and cannot gate this job, so branch protection keeps
it a required check for merge to `main`.

## Environment variables

Set automatically from `render.yaml`:

| Service  | Variable                  | Source / value                                    |
| -------- | ------------------------- | ------------------------------------------------- |
| backend  | `DATABASE_URL`            | from the managed DB (`connectionString`)          |
| backend  | `SECRET_KEY`              | generated by Render                               |
| backend  | `APP_CONFIG`              | `production` (debug off, dev server forbidden)    |
| backend  | `CORS_ORIGINS`            | the static-site origin                            |
| backend  | `RUN_INPROCESS_GENERATOR` | `true` — see "Data generator" below               |
| backend  | `WEB_CONCURRENCY`         | `1` (single web instance on free)                 |
| backend  | `SAMPLE_INTERVAL_SECONDS` | `10`                                              |
| frontend | `API_URL`                 | the backend's absolute `/api/v1/sensors` endpoint |
| frontend | `GRAFANA_URL`             | the Grafana dashboard URL (surfaces the in-app link) |
| grafana  | `DATABASE_URL`            | from the managed DB (`connectionString`)          |
| grafana  | `GF_AUTH_ANONYMOUS_ENABLED` | `true` — public read-only dashboard view        |
| grafana  | `GF_AUTH_ANONYMOUS_ORG_ROLE` | `Viewer`                                       |
| grafana  | `GF_SECURITY_ALLOW_EMBEDDING` | `true` — lets the SPA embed the dashboard in an iframe |
| grafana  | `GF_SECURITY_ADMIN_PASSWORD` | generated by Render                            |
| grafana  | `GF_SERVER_ROOT_URL`      | the Grafana service's external URL                |

The backend normalizes a legacy `postgres://` `DATABASE_URL` to `postgresql://`
automatically (`config.normalize_database_url`), since SQLAlchemy 2.x rejects
the old scheme. Grafana's entrypoint
([`deploy/grafana/render-entrypoint.sh`](../deploy/grafana/render-entrypoint.sh))
parses `DATABASE_URL` into the connection fields the datasource interpolates —
Render exposes the DB as a connection string, not as separate host/port.

### Matching the cross-service URLs

`CORS_ORIGINS`, `API_URL`, `GRAFANA_URL`, and Grafana's `GF_SERVER_ROOT_URL` are
written from the service **names**, so they resolve to
`https://sensor-app-{backend,frontend,grafana}.onrender.com`. If any name is
already taken in your Render account, Render appends a random suffix — edit those
values in `render.yaml` (or in the dashboard) to the real URLs and redeploy.

## Data generator (free-tier exception)

The canonical architecture runs the sensor generator as its **own process**
(`sensor_api.worker`), kept out of the app factory (CLAUDE.md §2.6). Render's
free tier has no background worker, so the Blueprint sets
`RUN_INPROCESS_GENERATOR=true` and the generator runs **in-process**, started
once inside the gunicorn request worker in
[`backend/gunicorn.conf.py`](../backend/gunicorn.conf.py) (`post_worker_init`)
as a daemon thread. It starts in the worker rather than the master to avoid the
fork-after-thread deadlock that would otherwise wedge the request workers.
`WEB_CONCURRENCY=1` keeps a single web worker, so there is exactly one
generator.

This is a deliberate, documented exception scoped to the free tier. On a paid
tier, drop `RUN_INPROCESS_GENERATOR` and add a separate worker service (the
existing `sensor_api.worker`) — see issue #15.

## How startup works

The backend image's default `CMD` is plain gunicorn (what compose uses).
Render overrides it with `dockerCommand: sh render-start.sh`, which runs the
Alembic migrations and then execs gunicorn — free instances have no separate
pre-deploy step, and `flask db upgrade` is idempotent, so running it on each
boot is safe.

## Free-tier caveats

- **Cold starts.** A free web service spins down after ~15 minutes idle; the
  next request wakes it (a few seconds). In-memory sensor state resets on each
  wake, but persisted readings survive in Postgres.
- **Time-limited database.** Render's free PostgreSQL expires after a fixed
  period; back up or upgrade before then.
- **Grafana cold starts + no disk.** The Grafana service also spins down when
  idle, so the "View charts" link cold-starts on the first hit. The free tier
  has no persistent disk, so users/settings reset on restart — fine here because
  the datasource and dashboards are baked into the image as provisioning and
  viewing is anonymous (read-only). The admin password resets to a freshly
  generated value on each deploy. A second free web service also draws on the
  account's shared free instance hours.
- **HTTPS only.** Render serves all services over HTTPS automatically.

## Verifying a deploy

```bash
curl https://sensor-app-backend.onrender.com/health    # {"status":"ok"}
curl https://sensor-app-backend.onrender.com/ready     # {"status":"ready"} once DB is reachable
curl "https://sensor-app-backend.onrender.com/api/v1/sensors?limit=5"
curl https://sensor-app-grafana.onrender.com/api/health   # {"database":"ok",...}
```

Then open the static-site URL and confirm the table populates (the generator
adds a reading every 10s), and follow the "View charts in Grafana" link — the
`Sensor Readings` dashboard renders the temperature/humidity and vibration
panels from the managed Postgres.

## Troubleshooting

- **Frontend shows an error / empty state.** Check the browser console for a
  CORS or 404 on `/api/v1/sensors`. Confirm `API_URL` (frontend) and
  `CORS_ORIGINS` (backend) point at the real, current URLs.
- **Backend unhealthy.** Check the deploy logs for the migration step; a failed
  `flask db upgrade` stops startup. Confirm `DATABASE_URL` is wired.
- **No new readings.** Confirm `RUN_INPROCESS_GENERATOR=true` on the backend and
  look for `in-process sensor generator started` in the logs.
- **Grafana shows no data / datasource error.** Check the Grafana deploy logs
  for the datasource health; confirm `DATABASE_URL` is wired from the DB and that
  the parsed `sslmode` is `require` (Render's managed PG mandates SSL). The "View
  charts" link 404s if `GRAFANA_URL` doesn't match the real Grafana URL.
